## v2.2.0 ‚Äî the FONT-METRIC / VERSION detector

The v2.1.0 self-check catches a feature's GLYPH being stripped (same width, wrong letterform). A sibling
class is the opposite: the SAME family at the SAME size renders to a CONSISTENTLY different WIDTH because the
two sides ship a DIFFERENT VERSION of the font. Unlike cv11 this IS a width difference ‚Äî so it is directly
measurable, and it is the ROOT CAUSE behind a height-differing line-count that the differ previously flagged
only as a symptom.

| # | What the differ was blind to | Why the prior checks missed it | The mechanical check (where) | Real diolog example |
|---|---|---|---|---|
| 32 | a homepage case-study body ("Beyond any single draft or reply‚Ä¶") with IDENTICAL declared font props on both sides ‚Äî Inter, 18px, weight 400, letter-spacing ‚àí0.36px, line-height 27.9px, container width 507px ‚Äî and the font APPLIES on both, yet LIVE wraps to 4 lines (height 112) while the TARGET wraps to 3 lines (height 84): "slightly larger on live" | every DECLARED-prop check matched (family/size/weight/letter-spacing identical) and the rendered-font DOM-span check said the family APPLIES on both, so nothing fired on the ROOT CAUSE. The height-aware line-count check flagged the SYMPTOM (4 lines/112 vs 3 lines/84) but never surfaced WHY: the exact text renders 636px on live vs 625px on target ‚Äî the target self-hosted a DIFFERENT VERSION of Inter (rsms v4, ~1.8% NARROWER than live's Google Inter v20) at the same size. The actionable signal is "same family + size, consistently different rendered WIDTH ‚áí different font version/metrics", which ‚Äî unlike cv11 (#31, same-width letterform) ‚Äî IS a width difference and so is directly measurable in-page | **FONT-METRIC / VERSION check** ‚Äî `analyze.js` `capture()` records each text node's EXACT-TEXT WIDTH (`exactW`): the node's own text (capped ~80 chars) laid out nowrap in an offscreen span using the element's EXACT computed font (family, weight, style, size, letter-spacing, word-spacing, font-feature-settings). ## The standing principles these encode

1. **Reference the LIVE rendered site, never a re-served scrape** (#6). A runtime-hydrated framework resolves a different variant off-origin. This alone caused several "I matched it exactly but it's wrong".
2. **`getComputedStyle(element)` is blind to pseudo-elements and to non-CSS layers** (#9, #10). Fold pseudo borders/shadows; capture media-layer children.
3. **Computed style can match while the RENDER differs** ‚Äî wrong glyph variant (cv11), variable-vs-static synthesis (#13), font-version width (#2), wrap balance (#5). Always carry **rendered geometry** (w/h/center-x), **glyph extent**, **line-count**, and **loaded faces** as ground truth, not just declared properties.
4. **Compare the SAME content on the SAME element type** (#7, #12). A node-boundary mismatch (dash-in-text, text-node-vs-button) makes a width/size comparison meaningless ‚Äî pair to the styled box and slice to the same string.
5. **Gaps live between elements, not only inside boxes** (#11, #15). Check sibling gaps and spacing accumulation, not just each box's own padding.
6. **When the human eye and a measurement disagree, the eye wins ‚Äî go LOOK / inspect the live element** (#9 was solved by the user reading `--border-color` in devtools). A confident wrong number is worse than no number; the fix is a better measurement, on the right node.
7. **Text-pairing alone is blind to NON-TEXT containers and to MISPAIRED repeated text** (#16, #19, #22, #17). A differ that only style-checks elements it matched BY TEXT never reaches a section/card/divider with no text of its own ‚Äî so ALSO match containers STRUCTURALLY (path ‚Üí geometry fallback) and compare their bg, full-bleed media layer, border (all sides, incl. `::after`), radius, shadow, and a page-wide hairline. And when the probe text is non-unique, DISAMBIGUATE by position before comparing, or every instance pairs to the first.
8. **`document.fonts.check` is necessary but NOT sufficient for "is the named face actually rendering"** (#20). A face can register as loaded yet not serve glyphs to rendered text (declared-vs-rendered drift). **The ground truth is a RENDERED-DOM metric, and it must be measured with a real DOM `<span>`, NOT canvas `measureText`** (v2.0.2). Canvas ignores `unicode-range` subsetting and does not replicate the browser's DOM font-matching cascade, so it cannot see a DOM-level fallback. Lay out a hidden span of a probe string twice ‚Äî `'<Named>', <generic>` vs bare `<generic>` ‚Äî for the same weight/style/size, against TWO distinct generics (monospace + serif); if the named width collapses to BOTH fallback baselines, the named family fell straight through to the fallback (`applies:false`). Await `document.fonts.ready` before probing. Compare the two SIDES' `applies` flags ‚Äî flag only a disagreement, never a per-node absolute.
9. **Illustrations are excluded as content-noise, but their POSITION is still a defect class** (#21). Compare media/mockup-card geometry RELATIVE to a stable section anchor (an eyebrow/heading), never by absolute y ‚Äî so page-wide vertical drift cancels and only a genuine in-section mispositioning fires. Position only, never internal content.
10. **A DECLARED + APPLIED OpenType feature can still have NO EFFECT ‚Äî only the rendered glyph SHAPE proves it took** (#31). `font-feature-settings:"cv11"` matching, the family applying (not a fallback), and the controlled span width being identical are ALL necessary but NOT sufficient: a subset woff2 that stripped the cvXX/ssXX glyph renders the default letterform at the SAME width, so every width / DOM-span / glyph-extent check is blind. The ground truth is a glyph-SHAPE comparison ‚Äî render the probe WITH vs WITHOUT the requested feature and pixel-compare; identical ‚áí ineffective. Pick the rasteriser by capability TEST (`'fontFeatureSettings' in ctx` ‚Üí in-page canvas; else a runner screenshot + pixel-diff of probe pairs), never by assumption. Compare the two SIDES' effectiveness ‚Äî escalate when the reference's feature takes and the target's does not ‚Äî and emit an honest "unresolved" note rather than a silent pass.
11. **The SAME font family at the SAME size can render to a CONSISTENTLY different WIDTH ‚Äî that is a different font VERSION, and unlike a glyph variant it IS measurable** (#32). When every declared font prop matches and the family applies on both sides, but a block "looks slightly larger on live" / wraps to a different line count, the root cause is usually a font-VERSION mismatch (the target self-hosts rsms Inter v4 ‚âà1.8% narrower than live's Google Inter v20 at the same size). Measure the EXACT text laid out nowrap in the element's exact computed font (family/weight/style/size/letter-spacing/word-spacing/feature-settings) on BOTH sides, take the width RATIO target/reference, and AGGREGATE per family ‚Äî a version mismatch is UNIFORM across the family, so the median ratio is a clean, low-noise signal; emit ONE finding per family above a small tolerance (0.7%, with a ‚â•70%-same-direction guard), never per element. The width-differing line-count is the SYMPTOM (a height-differing line-count is a REAL wrap divergence ‚Äî emit it confident/HIGH with the height delta, never bucket it as span-split noise); the width-ratio is the actionable root cause (swap to the matching font version). Contrast #31: a cvXX glyph stripped from a subset is the SAME width, so only the glyph SHAPE reveals it; a font-version drift IS a width difference, so it is measured directly.
