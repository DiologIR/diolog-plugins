# macOS 26 UI — Accumulated Learnings

> **What this file is.** The single, growing knowledge base of durable, generalisable patterns extracted from analysing real macOS UI screenshots. The **`macos-ui-analyst`** skill writes here (merging, never blindly appending); the **`macosify`** skill reads here as a third knowledge source alongside the bundled HIG (`../reference/hig/`) and `../reference/DESIGN.md`. It starts empty of real learnings — every entry below the divider is added from observed evidence.

## How to use this file

- **`macos-ui-analyst` (writer):** after each screenshot analysis, distil only the *durable, reusable* findings (a pattern that will hold for the next screenshot — not a one-off fact about this image) and **merge** them into the matching category below. Merging means: find an existing entry with the same semantic key → if corroborated, raise its confidence and add the new evidence; if it conflicts, keep both and mark `⚠ conflict`; if genuinely new, add a new entry. Never duplicate an existing rule; never let the file grow unbounded with near-duplicates. Keep entries terse.
- **`macosify` (reader):** load this file and prefer a high-confidence learning here over a guessed value, but **never** over an explicit HIG rule or a `DESIGN.md` token — precedence is **observed evidence in the target → HIG → DESIGN.md → this file**.

## Entry format

Each learning is one bullet:

```
- **<short rule>** — <the detail / value>. `origin: native|catalyst|ios-on-mac|swiftui|any` · `confidence: High|Med|Low` · `evidence: <app/screenshot, N sightings>` · `updated: YYYY-MM-DD`
```

Use the `origin` tag to record **which framework lineage** the pattern belongs to, so a Catalyst/iOS-on-Mac tell is never mistaken for a true-native rule. Tag a rule `any` only when it holds across lineages.

## Framework-origin reference (lineages to tell apart)

The analyst classifies every screenshot's likely lineage; the gold standard for *generation* is always **AppKit-native**. Quick tells (refine these as evidence accrues):

| Lineage | What it is | Native-feel | Common visual tells |
|---|---|---|---|
| **AppKit-native** | True Mac app (AppKit / native SwiftUI for macOS) | ✅ the target | 13pt body, 28/20pt compact controls, real menu bar, source-list sidebar with inset rounded selection, Liquid Glass floating chrome, concentric corners, pop-up/pull-down buttons |
| **Mac Catalyst** | UIKit iPad app bridged to AppKit | ⚠ iOS-derived | Oversized controls + padding, ~17pt body, iOS switch/segmented styling, iOS-style nav bars/back chevrons, plain sheets, weaker/auto-generated menu bar |
| **iOS-on-Apple-Silicon** ("Designed for iPad") | Unmodified iOS/iPadOS app in a resizable window | ❌ least native | Touch-sized 44pt controls everywhere, iOS tab bar at bottom/top, no real menu-bar depth, modal full-screen sheets, no pointer-precise affordances |
| **SwiftUI multiplatform** | One SwiftUI codebase | ✅/⚠ depends | Native when macOS modifiers/styles are used; iOS-like when iOS metrics/components leak through |

**Standing preference:** lean to AppKit-native. When a screenshot reads as Catalyst or iOS-on-Mac, record the *iOS-derived tell* AND its *AppKit-native correction* so `macosify` can convert toward native (e.g. 17pt body → 13pt; 44pt control → 28/20pt; iOS bottom tab bar → source-list sidebar / toolbar; full-screen modal sheet → window-anchored sheet).

## Window chrome & traffic lights

- **Coloured traffic lights = key/main window; grey = inactive (vibrancy dropped)** — read window focus straight from light colour before anything else; tie chrome vibrancy to focus. A parent window correctly dims its lights to grey while a sheet is up. `origin: native` · `confidence: High` · `evidence: Find My, Keynote, Mail, Music, Xcode, Home, Journal — active; Home/Journal sheet parents dimmed (10+ sightings)` · `updated: 2026-06-20`
- **Real traffic lights sit inside whatever chrome strip hosts the leading edge** (a sidebar head, a title bar) — even when the chrome's *fill/contents* are iOS-derived (Catalyst), the frame itself is genuine system chrome; judge lineage from the body, not the frame. `origin: any` · `confidence: High` · `evidence: Freeform-style board app, System Settings (Catalyst body in real chrome) (3 sightings)` · `updated: 2026-06-20`
- **Traffic-light left inset is the algorithmic concentric ratio of the title-bar height, never a hardcoded pad** — a tall unified toolbar yields a larger inset, a short title strip a small one; express via `.windowToolbarStyle`, never a constant. `origin: native` · `confidence: High` · `evidence: Calendar, Photos, Home, App Store, Mail (5+ sightings)` · `updated: 2026-06-20`
- **Unified-compact toolbar tell: the window title (or document name) shares the single toolbar row with the items** ⇒ `.windowToolbarStyle(.unifiedCompact)`; title stacked *above* the tool groups ⇒ `.expanded`/`.unified`. `origin: native` · `confidence: High` · `evidence: Calendar, Find My, GarageBand, Notes, Numbers, Keynote, App Store, Mail (8+ sightings)` · `updated: 2026-06-20`
- **A title carried as a large CONTENT heading (not a title-bar string) is a legitimate unified-compact variant** — a window can render its name as a Title-class heading at the top of content while the toolbar strip holds only traffic lights + trailing actions (Home "Rhodes Home"); when the view title lives in content, the toolbar title row is empty — don't infer `.unifiedCompact` vs `.unified` from a missing title (tag `<INFERENCE>`). `origin: native` · `confidence: Med` · `evidence: Home, Music, Photos (3 sightings)` · `updated: 2026-06-20`
- **A centred document title + draggable proxy icon + "— Shared"/"— Shared (Offline)"/"Edited" suffix is the AppKit `NSDocument` titlebar** — bind `window.representedURL` + `isDocumentEdited`; the status suffix is live iCloud/dirty state, not a hardcoded string; primary name primary-label, the suffix in a dimmer secondary tier. `origin: native` · `confidence: High` · `evidence: Keynote, Numbers iWork document windows (4 sightings)` · `updated: 2026-06-20`
- **macOS 26 windows float on the desktop with a large concentric corner radius + a soft real drop shadow; the surrounding pure black/white is the desktop, not the window fill** — record the OBSERVED radius, never assume a 16pt constant (macOS 26 radii are fragmented across frameworks); drive nested children with `.rect(cornerRadius: .containerConcentric)`. `origin: native` · `confidence: High` · `evidence: Calendar, Photos, Music, Mail, Xcode + every report's corner note (15+ sightings)` ⚠ see Open questions for the radius-constant conflict · `updated: 2026-06-20`
- **macOS Settings/preferences windows dim the minimize + zoom lights, keeping close active** — a strong "this is a `Settings { }`/⌘, window" tell, because the window reopens instantly with ⌘,; expect a flatter/greyer middle+right light. `origin: native` · `confidence: High` · `evidence: journaling-app Settings, System Settings panes (2 sightings)` · `updated: 2026-06-20`
- **An alert/dialog has NO traffic lights, toolbar, or title bar** — it is dismissed only by a button (or Esc / ⌘-Period); absence of window chrome is the correct alert signature, not a missing feature. `origin: native` · `confidence: High` · `evidence: Keynote "Update to Collaborate" alert (1 sighting)` · `updated: 2026-06-20`
- **A system Open/Save panel presents as a SHEET (no own traffic lights) over the document window** — absence of traffic lights + a rounded card over a dimmed parent = sheet presentation, not a standalone window. `origin: native` · `confidence: Med` · `evidence: Keynote Open panel (1 sighting)` · `updated: 2026-06-20`
- **Filename gotcha (process): macOS screenshot filenames embed U+202F (narrow no-break space, bytes e2 80 af) before "am/pm"** — `ls`/`grep` show the entry but a literal-ASCII-space path fails `cp`/`stat`/Read; glob the timestamp (`*pm.png`) or match the real bytes, then read the copy. (One report saw a U+00A0 NBSP instead — either way, never trust a hand-typed space.) `origin: any` · `confidence: High` · `evidence: ~20 reports' file-access notes` · `updated: 2026-06-20`

## Liquid Glass & materials

- **Golden Rule: glass belongs ONLY to the floating functional layer (toolbar, sidebar, floating panels/popovers/menus, alerts); the content layer stays opaque** — never apply `.glassEffect()` to a document canvas, spreadsheet/calendar grid, code editor, gallery/board card, media thumbnail, chat bubble, audio/waveform card, settings/form row, dialog body, or list/table. This is the single most-repeated rule. `origin: any` · `confidence: High` · `evidence: nearly every report (40+ sightings)` · `updated: 2026-06-20`
- **In dark mode, large glass (sidebars, toolbars, inspector panes) reads as near-opaque graphite and does NOT light/dark-flip** — subtle dark translucency over uniformly dark content is correct, not a missing-glass defect; only small chrome flips with the background. A flat-reading dark sidebar is therefore ambiguous between Regular glass and Reduce-Transparency — tag `<INSUFFICIENT_EVIDENCE>` rather than asserting either. `origin: native` · `confidence: High` · `evidence: App Store, Photos, Calendar, Numbers, Home, Mail (8+ sightings)` · `updated: 2026-06-20`
- **Default an ambiguous dark floating-chrome material to Regular Liquid Glass (never Clear)** — dark compresses visible translucency so a near-opaque toolbar is still Regular glass + scroll-edge; a text/control-bearing popover takes Regular even over bold media. Clear is reserved for small controls over bold media. Never set a custom toolbar background. `origin: native` · `confidence: High` · `evidence: Calendar, Keynote popovers, Mail, Notes (6+ sightings)` · `updated: 2026-06-20`
- **Scroll-edge effect = the bright translucent top/bottom fade where content slides UNDER a Regular-glass toolbar/composer** — it is the BAR's edge effect (lensing/specular), not a glass card behind the content and not row-opacity styling; encode it on the toolbar node (`ScrollEdgeEffectStyle`), keep the content opaque. `origin: native` · `confidence: High` · `evidence: Calendar, App Store, Photos, Music, Messages, Xcode (7+ sightings)` · `updated: 2026-06-20`
- **A glass popover LENSES (samples a dimmed version of) the content behind it, not a fixed grey** — a dark Regular-glass popover over a navy slide shows a dimmed-navy interior (e.g. doc `rgb(18,51,95)` → interior `rgb(15,34,61)`); a flat opaque grey card with no specular edge / no background bleed is a non-native (legacy or web) tell. `origin: native` · `confidence: High` · `evidence: Keynote "Add a Slide" / chart popovers (3 sightings)` · `updated: 2026-06-20`
- **Proximal capsule/symbol controls sharing ONE continuous refractive edge ⇒ a `GlassEffectContainer` (per-child `glassEffectID`), not N separately-bordered glass pills** — adjacent glass tiles/buttons are the glass-on-glass risk point; wrap them. Two-up action tiles (Play Sound + Directions), trailing +/••• clusters, Safari's leading cluster, Mail's reply/archive/move groups, Keynote's insert cluster all read as containers. `origin: native` · `confidence: Med` · `evidence: Find My, Home, Safari, Mail, Keynote, Music mini-player (6 sightings)` · `updated: 2026-06-20`
- **On-glass recessive controls are translucent FILLS/vibrancy ON the one glass layer, never nested glass** — an accent-tinted selector or a neutral alert button on a glass surface is a vibrant fill, not a second backdrop-filter; avoids glass-on-glass. `origin: native` · `confidence: High` · `evidence: Keynote category pill, Keynote alert buttons (2 sightings)` · `updated: 2026-06-20`
- **Parent-dim ≠ a glass scrim** — the darkened/tinted canvas behind a centred dialog/sheet is the standard modal parent-dim; faint legible text/shapes behind a modal are the DIMMED PARENT bleeding through Regular glass, not a second glass layer or a Clear-glass dimming layer. `origin: native` · `confidence: High` · `evidence: Keynote dialog/alert, Home/Journal sheets (4 sightings)` · `updated: 2026-06-20`
- **Content-layer cards carrying their own gradient/photo/map imagery are NOT Liquid Glass** — they don't sample the desktop, so a coloured Insights/Places card, a wallpaper tile, an accessory tile, or an audio-clip card is an opaque content fill, not `.glassEffect()`; don't let a coloured card trip the glass audit. `origin: any` · `confidence: High` · `evidence: Journal cards, Home tiles, Numbers chart swatches (5 sightings)` · `updated: 2026-06-20`
- **Wallpaper visible through a window's top/bottom bands ≠ glass on content** — an aurora/desktop showing at a window's transparent chrome bands is the floating toolbar/composer's transparency + scroll-edge, not a glass transcript background; the content layer (bubbles, rows) stays opaque. Mail's sidebar showing the desktop through its rounded corner is `under-window` vibrancy (the sidebar is translucent material). `origin: native` · `confidence: High` · `evidence: Messages over aurora desktop, Mail sidebar corner show-through (2 sightings)` · `updated: 2026-06-20`
- **The macOS 26 Music mini-player is a centred floating Liquid-Glass CAPSULE pill over the content's bottom edge** (transport ‖ art+LCD+scrubber ‖ queue/AirPlay/volume), not a docked opaque bottom bar — model as a `GlassEffectContainer` + `.glassEffect(.regular, in: .capsule)`, inner buttons/art as opaque/vibrant fills (no nested glass). `origin: native` · `confidence: High` · `evidence: Apple Music — Songs (1 sighting)` · `updated: 2026-06-20`
- **Reduce-Transparency solidifies the materials + drops the blur; Increase-Contrast firms the hairlines** — most captures read `accessibility_state: standard` (translucency present, soft hairlines); flag the solidified/firmed read only when evidence supports it. `origin: native` · `confidence: Med` · `evidence: Find My, multiple a11y-state reads (3 sightings)` · `updated: 2026-06-20`

## Layout, spacing & geometry (grid, concentric corners)

- **Concentric corners: child radius = parent radius − padding; record observed, never assume a constant** — sheet card > inner grouped cards > tiles; toolbar items concentric with the bar; thumbnail radius < card radius. macOS 26 radii are fragmented across AppKit/Catalyst/SwiftUI, so emit `.rect(cornerRadius: .containerConcentric)` + a measured outer value, not a literal 16pt. `origin: native` · `confidence: High` · `evidence: GarageBand sheet, Keynote, Mail, every corner note (12+ sightings)` ⚠ see Open questions · `updated: 2026-06-20`
- **macOS controls snap to the 8pt grid at desktop density** — ~16–20pt internal padding, ~16pt card gaps, leading icon→label ~8pt, section insets ~16–20pt; row vertical rhythm padded to reach the 44pt hit target while the *visible* control stays ~20–28pt. `origin: native` · `confidence: Med` · `evidence: GarageBand sheet, System Settings, Mail (4 sightings)` · `updated: 2026-06-20`
- **Compact desktop density (~13pt body, ~20–28pt controls, ~24–28pt rows) is the single strongest AppKit-native-vs-iOS discriminator** — the absence of 44pt touch rows / 17pt body / inflated padding rules out Catalyst & iOS-on-Mac. Check body size first to separate genuinely-desktop apps from iOS-derived ones, even on web-wrapped apps. `origin: native` · `confidence: High` · `evidence: Calendar, GarageBand, Home, Numbers, App Store + most reports (15+ sightings)` · `updated: 2026-06-20`
- **Four visible columns ≠ a 4-pane split** — a 3-pane `NavigationSplitView` (sidebar · content · detail) plus a trailing `.inspector(isPresented:)` reads as four columns but is one window/title/toolbar; model the trailing pane as an inspector, not a 4th split pane. `origin: native` · `confidence: Med` · `evidence: Xcode Organizer, Xcode IDE (2 sightings)` · `updated: 2026-06-20`
- **Split-view panes meet at a 1pt hairline divider, not a glass seam or a heavy border** — the inspector/content/sidebar separation is a thin `separator`, and content tones may step (sidebar darker than content). `origin: native` · `confidence: High` · `evidence: Notes inspector, Xcode, Calendar dark levels (4 sightings)` · `updated: 2026-06-20`
- **When the window top is cropped, leave toolbar-style `<MISSING_DATA>` — never hardcode the traffic-light inset** — without the titlebar height you cannot derive `.unifiedCompact` vs `.expanded`; record the gap, don't invent it. `origin: any` · `confidence: High` · `evidence: cropped Keynote chart-picker (report-34) (1 sighting)` · `updated: 2026-06-20`

## Typography (SF Pro, sizes, weights, tracking)

- **The UI face is SF Pro via `-apple-system`; macOS Body is 13pt (Caption ~11pt, gutter/footnote ~10pt) — NOT iOS 17pt** — no web fonts bundled; resolve to named text styles, never hardcoded px. iOS 17pt body leaking into a Mac window is the loudest lineage tell. `origin: native` · `confidence: High` · `evidence: nearly every report (20+ sightings)` · `updated: 2026-06-20`
- **The note-list / source-list row three-tier text maps straight to the label hierarchy** — bold/semibold title = `label`/Body-Semibold, preview/metadata = `label-secondary`, relative timestamp = `label-tertiary`; getting these alphas right is most of the native feel of a list cell. `origin: any` · `confidence: High` · `evidence: Notes/notes-app rows, Mail, Music, Home tiles (6 sightings)` · `updated: 2026-06-20`
- **A two-tier title (bold primary + secondary-label suffix) is a native idiom** — `June` bold + `2026` secondary in a toolbar; document name primary + "— Shared" secondary. `origin: native` · `confidence: Med` · `evidence: Calendar title, Keynote/Numbers proxy titles (3 sightings)` · `updated: 2026-06-20`
- **Timecodes / elapsed readouts use monospaced (tabular) digits** — large `00:00.00`-style numerals are SF Mono / `.monospacedDigit()` so glyphs don't jitter as numbers tick. `origin: native` · `confidence: High` · `evidence: Notes recording timecode (2 sightings)` · `updated: 2026-06-20`
- **Document/canvas content type is NOT UI type** — slide title/subtitle/body, chart data, and email body are document content (large display sizes, app theme colours); don't resolve them to macOS named text styles or chrome tokens. `origin: any` · `confidence: High` · `evidence: Keynote/Numbers canvas, Mail HTML email (4 sightings)` · `updated: 2026-06-20`
- **A scrimmed/dimmed parent fades its lower-tier labels toward `label-tertiary`** — apparent low-contrast text under a sheet is the dimming overlay, not a real contrast token. `origin: any` · `confidence: Med` · `evidence: Journal footnote under sheet scrim (1 sighting)` · `updated: 2026-06-20`

## Colour & tokens (label tiers, system palette, accent)

- **Bind selection/focus/primary-action emphasis to `controlAccentColor` (CSS `accent-color`) — never hardcode blue** — the app accent can be non-blue (Home orange, Journal indigo/violet, Music red, Numbers green), and a uniformly-tinted UI (selected-row fill + sidebar glyphs + content glyphs all one hue) signals the USER's system accent, not a brand colour. Resolve to the token, record the observed hue. `origin: native` · `confidence: High` · `evidence: Home orange, Journal violet, Music red, Numbers green, Home Automation orange (8+ sightings)` · `updated: 2026-06-20`
- **The six-tier label model (primary ~100%/85% white dark, secondary ~55%, tertiary ~25%) carries most of the dark-mode feel** — primary for titles/values, secondary for metadata/captions/section headers, tertiary for placeholders/dim ticks. Resolve to `label`/`label-secondary`/`label-tertiary`, never raw greys. `origin: native` · `confidence: High` · `evidence: most reports (15+ sightings)` · `updated: 2026-06-20`
- **Dark-mode chrome is graphite, not pure black, with the sidebar a half-step off the content base** — author dark independently (NOT an inversion of light); window/content ~`#1E1E1E`-class, surfaces ~`#2C2C2E`/`#3A3A3C`, sampled Calendar content `#25292c` vs sidebar `#1d2022`. Spread tonal levels for pane separation. `origin: native` · `confidence: High` · `evidence: Calendar, GarageBand, App Store, Mail, Xcode + most dark reports (12+ sightings)` · `updated: 2026-06-20`
- **Per-item identity colours map to the 12-hue SYSTEM PALETTE, not the single accent** — calendar tints, event pills, chart series, sidebar collection glyphs, binary-image badges; these are the sanctioned "fixed colour used sparingly/meaningfully" case, independent of `controlAccentColor`. Don't read every coloured element as "the accent." `origin: native` · `confidence: High` · `evidence: Calendar checkboxes/pills, Numbers charts, Mail VIP yellow, Xcode badges (8 sightings)` · `updated: 2026-06-20`
- **An app accent (or object/sheet-tab tint) can differ from the user's `controlAccentColor`** — Numbers' green sheet-tab + selected segment, GarageBand's green selection ring, Music's red, HomeKit's orange are app-semantic/brand tints; map to the matching system-palette hue but flag it as app-semantic, and for a generic clone still bind selection to `controlAccentColor`. `origin: native` · `confidence: Med` · `evidence: Numbers green, GarageBand green ring, Music red, Home orange (5 sightings)` · `updated: 2026-06-20`
- **Status/now/today colour is paired with a label or glyph/shape, NEVER colour alone** — green "Now"/"Connected", orange "Lost"/warning triangle, red record/playhead/today, "No Response" + `exclamationmark.triangle.fill`; today is a FILLED accent (system-red) disc behind the day numeral; record/playhead red is status colour and must NOT bind to the accent. `origin: native` · `confidence: High` · `evidence: Calendar today/now, Find My status, Notes record, Home tiles, System Settings Wi-Fi (8 sightings)` · `updated: 2026-06-20`
- **The single primary action is the one accent-tinted control; tint nothing else** — "when every element is tinted, nothing stands out." Tint only the selected sidebar row / the one prominent button / links; chrome stays neutral graphite, content carries saturated colour (chart data, artwork). `origin: native` · `confidence: High` · `evidence: GarageBand link+Continue, Home, Keynote dialog, Numbers (6 sightings)` · `updated: 2026-06-20`
- **The text-insertion caret is often the only accent pixel** — bind the caret to `controlAccentColor`; don't infer the whole window's accent from a record/status red elsewhere. `origin: native` · `confidence: High` · `evidence: Notes editor caret vs red record state (1 sighting)` · `updated: 2026-06-20`
- **`::selection` is accent-tinted; the accent flows to selection rings, the selected sidebar row, and the focus ring** — System Settings' "Colour" swatch strip IS the system accent picker. `origin: native` · `confidence: High` · `evidence: System Settings Appearance (2 sightings)` · `updated: 2026-06-20`
- **A page/content-area title is normally primary `label` (Bold), NOT accent-tinted** — a bright-accent large heading (e.g. blue "All Boards") is a brand choice to flag; reserve the accent for selection/primary action/links. `origin: native` · `confidence: Med` · `evidence: Freeform-style board gallery "All Boards" heading (2 sightings)` · `updated: 2026-06-20`

## Controls (buttons, fields, toggles, segmented, pickers, lists/tables, sidebars, toolbars)

- **Source-list selection = a flat inset rounded-rect fill with accent-tinted text/glyph, NOT a full-bleed bar and NOT a glossy saturated accent capsule** — the single most-repeated native tell; the same flat grammar spans sidebar AND content table/list. The selected row reads "chosen", not "pressed button". `origin: native` · `confidence: High` · `evidence: Calendar, Find My, Photos, Mail, Music, App Store, Xcode, Home (15+ sightings)` ⚠ System Settings uses a stronger fill — see Open questions · `updated: 2026-06-20`
- **Sidebar/source-list section headers are system-font, semibold, secondary colour, SENTENCE/title case — NEVER tracked uppercase** — "iCloud"/"Diolog"/"Favourites"/"Smart Mailboxes"/"Pinned"/"Reports"; the #1 sidebar authenticity tell and the loudest web/Catalyst correction. Settings sidebars may group by spacing with no header at all. `origin: native` · `confidence: High` · `evidence: Calendar, Photos, Mail, Keynote Open panel, System Settings, Xcode (12+ sightings)` · `updated: 2026-06-20`
- **Sidebar trailing counts render as quiet secondary numerals / `.badge(n)`, reserved for unread/pending counts only** — "5" Updates, "272", "17/17/1/0"; not a bright saturated pill, and never repurposed for non-count state. `origin: native` · `confidence: High` · `evidence: App Store, Photos, Calendar, Freeform-style gallery, Journal (6 sightings)` · `updated: 2026-06-20`
- **Sidebar row glyphs default to monochrome SF Symbols tinted to `controlAccentColor` (the selected row recolours both glyph and label); fixed per-row colours are allowed only sparingly to clarify meaning** — Mail's VIP yellow / flag red is the sanctioned case; blanket fixed colours that override the user's accent everywhere is the anti-pattern. `origin: native` · `confidence: High` · `evidence: Calendar, Mail, Photos, Home (6 sightings)` · `updated: 2026-06-20`
- **Pop-up button tell: a (often accent-tinted) control whose LABEL updates to the current value, with a DOUBLE up/down chevron** — `NSPopUpButton`/`Picker(.menu)`; choose-one mutually-exclusive value. "125%"/"Colour LCD"/"Widescreen (16:9)"/"Always"/"2D Graphs". `origin: native` · `confidence: High` · `evidence: GarageBand, Numbers, Keynote, System Settings, Safari (10+ sightings)` · `updated: 2026-06-20`
- **Pull-down button = a SINGLE down chevron + a STATIC title (an actions menu), distinct from a pop-up** — "Notes ▾", Safari's "Personal" Tab Group, Music's "New", a zoom menu. A trailing single right/down chevron on a NAV row is instead a disclosure-into-detail affordance, not a pop-up. `origin: native` · `confidence: High` · `evidence: Safari Tab Group, Music New, System Settings list rows (5 sightings)` · `updated: 2026-06-20`
- **Stepper tell: split up/down chevrons ALWAYS paired with an editable value field, in the window body (never the toolbar)** — `NSStepper`+`NSTextField` for precise numeric entry ("120", "4/4", "5.0 s", "490 pt", "11 pt"); a bare valueless stepper is the anti-pattern. Use a slider for approximate ranges, a stepper+field for exact values. `origin: native` · `confidence: High` · `evidence: GarageBand, Keynote, Numbers inspectors (6 sightings)` · `updated: 2026-06-20`
- **Pick the control by MEANING: radio group for mutually-exclusive sets, checkbox (square+checkmark) for independent/minor settings, switch for an emphasized/group-governing toggle, pop-up for value selection** — a green `UISwitch` on every settings row is the iOS-derived tell; the AppKit trio is checkbox/radio/pop-up. Checkboxes are leading-aligned, indented to show dependency, and live in the body. `origin: native` · `confidence: High` · `evidence: Numbers settings, System Settings, Keynote/Numbers inspectors, Xcode signing (8 sightings)` · `updated: 2026-06-20`
- **A switch is shorter/compact on Mac (`NSSwitch` ~22pt) and binds to `controlAccentColor`; an off switch parks the knob leading on a grey track** — a tall (~31pt) bright-tinted capsule track + large knob is the iOS `UISwitch` (Catalyst tell); read on/off from knob position + track tint, never colour alone. `origin: native` · `confidence: High` · `evidence: Find My, System Settings, Home (Catalyst contrast) (5 sightings)` · `updated: 2026-06-20`
- **A segmented control is for switching VIEWS/OPTIONS inside a toolbar/inspector/popover, NOT for main-window navigation (use a tab view/sidebar)** — legitimate: Calendar's Day/Week/Month/Year (the Apple-blessed date-interval exception), Find My People/Devices/Items, grid/list view toggle, file-browser view-mode, inspector Style/Text/Arrange, Trackpad sub-tabs, 2D/3D/Interactive popover scope. ≤7 equal-width text segments; the default macOS selected segment is a NEUTRAL raised fill (an accent-filled capsule is an app brand choice). `origin: native` · `confidence: High` · `evidence: Calendar, Find My, Numbers/Keynote inspectors, Keynote popover, System Settings, Xcode (10+ sightings)` · `updated: 2026-06-20`

- **⚠ conflict — macOS System Settings lineage: Catalyst vs native/SwiftUI gold-standard.** Two reports classify it `catalyst` (iOS-Settings layout ported via Catalyst — inset-grouped lists, `UISwitch`, accent-capsule selection, coloured icon chips, back-chevron nav are iOS tells to correct); seven classify it `native`/`swiftui` (the canonical macOS-26 reference: `NavigationSplitView` + opaque grouped form, sidebar-top search, pop-up double-chevrons, sentence-case headers, 13pt density, no iOS metrics).
