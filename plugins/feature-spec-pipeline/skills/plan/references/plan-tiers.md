# Plan size tiers, templates, and quality bar

The plan should be proportional to the task. A trivial UI tweak does not need the same structure as a cross-cutting feature.

## Avoid over-engineering

Don't pad the plan. A bug fix or simple layout change doesn't need an Edge Cases section, a Requirements Traceability table, or a multi-section requirements list. Only include sections where the content is non-obvious and actionable. **A 10-line diff should produce a ~30-line plan, not a 260-line plan.** If a section would just say "no changes required", omit it.

- No "Edge Cases" for anything without genuine edge cases.
- No "Testing" section when manual runtime verification is sufficient and already covered in a Verify step.
- No "Requirements Traceability" table unless the plan has >10 requirements across multiple layers.
- No copy-paste of the spec's feature description — summarize intent.
- No multi-step verification when a single step with bullets covers the same signal.
- No ceremonial fields ("Confirm no orphaned imports" / "Fulfils: non-regression" is ritual, not planning).

## Classify the tier (state it once, consistently)

Write `**Plan size:** [tier]` at the top of the plan and use the same tier name in the console summary.

- **Trivial** — single-file copy change, one-line CSS tweak, renaming a constant, deleting dead code. Target **<50 lines**.
- **Small** — single-file UI tweak, pure layout move, a 2–3 file change with no backend/data-model impact, a simple bug fix localised to one component. Target **<120 lines**.
- **Standard** — multi-file change, new component + its wiring, a new tool/endpoint registered in an existing factory/router, a new field on an existing type. Touches backend and frontend, but only through existing extension points. Target **<350 lines**.
- **Large** — cross-cutting work introducing **at least one new unit of architecture**: a new NestJS module, a new Mongoose schema/collection, a new page/route, or genuinely >15 requirements across 3+ layers. Target **<700 lines**.

**Standard ↔ Large tie-breaker.** Large requires at least one of: a new NestJS module (not editing one), a new Mongoose schema/collection (not adding a field), a new page/route (not adding a component into a page), or >15 requirements across 3+ layers. Touching backend + frontend both is **not** sufficient for Large — a cross-cutting change that only extends existing modules/schemas/pages is **Standard**.

## Codebase investigation (proportional to tier)

For the elements the spec references: locate each with Glob/Grep and **read the file** (don't stop at the path); for data features trace UI → query/mutation → resolver → service → schema; identify the closest existing analogue and follow it (don't invent patterns); map imports/shared utilities/types/components touched; check naming (UI labels vs route/component names) and flag ambiguity; review the current GraphQL schema if the feature touches the API; look for existing TODOs/stubs/partials. Skip steps that don't apply (a CSS-only change needs no schema trace). For Standard/Large, parallelize this across subsystems via the `Workflow` tool, then synthesize.

## Quality bar

A good plan at any tier: names specific file paths, functions, types, components to create/modify; points at a specific existing analogue as the template; orders steps by dependency (schema → types → queries → components); is specific about UI/UX states (loading/empty/error) that are genuinely non-obvious; includes verification targeted to what matters. A bad plan: vague language ("update the component"), references files that don't exist, invents patterns, pads with empty sections, transcribes the description.

## Shared header (all tiers)

```markdown
# Implementation Plan: <ID>
<!-- Generated by /plan -->

**Title:** [Title from the spec header]
**Spec ID:** <ID>
**Spec:** docs/specs/spec-<ID>.md
**Plan size:** [Trivial | Small | Standard | Large]
```

## Template — Trivial / Small (<50 / <120 lines)

```markdown
## Task
[2–4 sentences on what to change and why. Capture intent; don't transcribe the spec.]

## Files
- `[path]` — [one-line role]

## Changes
1. [Concrete change with exact insertion point and the analogue to mirror.]
2. …

## Reference
[Closest existing analogue — "mirror the Card/CardHeader/CardBody shape at `…:340`".]

## Verify
- [Browser/runtime check.]
- [Responsive/state check if applicable.]
- [`pnpm typecheck` / `pnpm lint` / zero console errors, as applicable.]

## Out of scope
- [Only if there's a real thing the reader might assume is in scope. Omit otherwise.]
```

Do NOT add at this tier: Edge Cases, Testing, Requirements Traceability, Prerequisites, Acceptance Criteria, multiple verification steps.

## Template — Standard (<350 lines)

```markdown
## Task
[3–6 sentences on intent, scope, user-facing goal.]

## Approach
[One paragraph: which layers are touched, which analogue you follow.]

## Reference implementation
[Specific existing feature to model after.]

## Prerequisites
- [Real setup only: migrations, codegen, env vars. Omit the section if none.]

## Steps

### 1. [Step title]
- **File:** `[exact path]`
- **Action:** Create | Modify
- **Details:** [Specifics.]
- **Reference:** [Analogue to follow.]
- **Verify:** [How to confirm this step.]
- **Fulfils:** [Short label — omit if traceability adds no value here.]

### 2. …

## Edge cases
- [ONLY genuine edge cases needing code. Omit if none.]

## Verify
- [One block of bullets: browser/runtime, responsive, console errors, type/lint gates. One step beats four.]

## Out of scope
- [Real things a reader might assume are in scope.]
```

Do NOT add a Requirements Traceability table at this tier — the per-step **Fulfils** line carries the mapping.

## Template — Large (<700 lines)

Use only when the plan genuinely has >10 requirements across multiple layers. Start with the Standard template, then add:

```markdown
## Requirements & Decisions
[Consolidate all requirements. Sources: the spec's feature description, human answers, the prior triage Assumptions block.]

### Functional
1. [Requirement with source attribution if from the triage section.]

### UI/UX
1. [Location on page, states, interaction behaviour.]

### Constraints & Decisions
- [Constraints from the spec; decisions locked by the triage Assumptions block.]

## Acceptance Criteria
- [ ] [Testable done criterion.]

## Testing
- [What tests to write and where. Map back to acceptance criteria.]

## Requirements Traceability
| Requirement | Step(s) |
|---|---|
| Functional #1 | Step 3, Step 7 |
```

Even at Large tier, omit individual sections that would be empty.
